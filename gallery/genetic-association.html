Genetic Association (R + TS)

The same forest plot, two ways — both rendered live, from R and from TypeScript

When tabviz was just an R package, “the JS code” meant “the bundled htmlwidget runtime” — opaque to consumers, only useful as a black box behind the R API. As of 0.2.0, the JS side is a first-class authoring target: the same tabviz() / col_*() / viz_*() / theme_*() API mirrored in TypeScript, available from npm as @tabviz/core.

This page renders the same forest plot twice on the same page — once via R, once via TypeScript loaded over the wire from esm.sh. The pixels should match.

Data: genetic associations with cardiovascular events

A small GWAS-style result table — SNP-level effect estimates from a polygenic-risk-score study. Rows are SNPs; columns carry the locus name, allele frequency, effect size + CI, sample size, and a p-value. A forest plot on the right shows the effect estimates aligned with the table.

R: tabviz(...) rendering the live widget

Code 
tabviz(
  snps,
  label = "snp",
  label_header = "SNP",
  theme = web_theme_nejm(),
  columns = list(
    col_text("gene", header = "Gene"),
    col_numeric("maf", header = "MAF", decimals = 2),
    col_interval("effect", "lower", "upper", header = "Effect (95% CI)"),
    col_n("n_studies", header = "Studies"),
    col_pvalue("p_value"),
    viz_forest(
      point = "effect", lower = "lower", upper = "upper",
      axis_label = "Effect estimate",
      scale = "linear",
      null_value = 0
    )
  ),
  title = "Genetic associations with cardiovascular events",
  caption = "Effect = log odds ratio per allele; CI = 95%."
)

TypeScript via @tabviz/core, rendered in the browser

Below is the same plot built from the TypeScript-mirror of the R API and mounted by createTabviz — executed live in this page (no R involved). The package is loaded from esm.sh; Svelte’s optional peer dep is resolved transitively.

Code 
core = import(`https://esm.sh/@tabviz/core@${tabviz_core_version}?deps=svelte@${svelte_version}`)

// Quarto exposes `snps_in` as a column-oriented table; transpose to rows.
rows = transpose(snps_in)

styleHref = `https://esm.sh/@tabviz/core@${tabviz_core_version}/dist/style.css`
// Inject the @tabviz/core stylesheet once per page.
_styleInjected = {
  if (!document.querySelector(`link[href="${styleHref}"]`)) {
    const link = document.createElement("link");
    link.rel = "stylesheet";
    link.href = styleHref;
    document.head.appendChild(link);
  }
  return true;
}

tsWidget = {
  const { tabviz, createTabviz,
          colText, colNumeric, colInterval, colN, colPvalue,
          vizForest } = await core;
  _styleInjected;

  const spec = tabviz({
    data: rows,
    label: "snp",
    labelHeader: "SNP",
    theme: "lancet",
    columns: [
      colText({ field: "gene", header: "Gene" }),
      colNumeric({ field: "maf", header: "MAF", decimals: 2 }),
      colInterval({
        point: "effect", lower: "lower", upper: "upper",
        header: "Effect (95% CI)",
      }),
      colN({ field: "n_studies", header: "Studies" }),
      colPvalue({ field: "p_value" }),
      vizForest({
        point: "effect", lower: "lower", upper: "upper",
        axisLabel: "Effect estimate",
        scale: "linear",
        nullValue: 0,
      }),
    ],
    title: "Genetic associations with cardiovascular events",
    caption: "Effect = log odds ratio per allele; CI = 95%.",
  });

  const host = document.createElement("div");
  host.style.minHeight = "320px";
  createTabviz(host, spec);
  return host;
}

import {
  tabviz, createTabviz,
  colText, colNumeric, colInterval, colN, colPvalue,

  theme: "lancet",

    colPvalue({ field: "p_value" }),

What’s the same, what’s different?

The argument names, defaults, and shape of the returned spec are identical:

  • R: col_interval("effect", "lower", "upper") ↔︎ TS: colInterval({ point: "effect", lower: "lower", upper: "upper" })
  • R: snake_case kwargs ↔︎ TS: camelCase keys
  • R: theme = web_theme_nejm() ↔︎ TS: theme: "lancet" (or theme: themeLancet())

R is positional-friendly (col_* helpers take field as the first positional arg); TS is purely named-argument because it’s a single object literal. Beyond that, the helpers are mechanical 1:1 translations — change one to fit the other by remembering snake-case ↔︎ camelCase and keyword-vs-positional.
