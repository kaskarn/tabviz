Post-0.2 retrospective: bilingual authoring + closing the OKLab gap

Posted: 2026-05-19 Author: Claude (via Antoine) Status: Closing notes for the 0.2.x line — @tabviz/core@0.2.0 through 0.2.1 and the R companion tabviz@0.31.2. Companion to - docs/dev/phase0-devblog.md (pre-split paydown) - docs/dev/post-publish-devblog.md (Phase 1 through the npm publish) - docs/dev/r-ts-parity-notes.md (per-helper R↔︎TS parity tracker)

The post-publish devblog ended with @tabviz/core@0.1.1 on npm — a runtime-only package. This entry covers what happened next: the JS side becoming a first-class authoring target, the OKLab precision gap we discovered while porting the theme cascade, and three rounds of opportunistic R-side cleanup that fell out of the parity work. By the end the npm package shipped a 1:1 TypeScript mirror of tabviz() / col_*() / viz_*() / theme_*(), the View Source panel emitted compact builder code instead of 200-line JSON dumps, and the Quarto docs gallery had two pages live-rendering the same plot from both R and TypeScript on the same page.

What we were trying to do

The 0.1.x line proved you could load @tabviz/core from npm and mount a server-built WebSpec into a DOM node. Useful for consumers who had R generating specs, but it left the JS side as a consumer of payloads constructed elsewhere. The natural next question — “can I author a spec from TypeScript without R in the loop?” — required mirroring the R authoring surface (tabviz(), col_text(), viz_forest(), web_theme_lancet(), the spec modifiers) in TS.

Three goals for 0.2.0:

  1. TS authoring builders mirroring R 1:1. colText({ field: "x" }) should produce the same wire shape as col_text("x"), with matching argument names + defaults (snake_case ↔︎ camelCase translation, but nothing else surprising).
  2. A real theme story JS-side. The R-side cascade resolver (R/utils-theme-resolve.R, ~680 lines of 3-tier OKLCH derivation) needed a TS counterpart so { theme: { extend: "lancet", overrides: { inputs: { primary: "#..." } } } } could re-derive every Tier 2/3 leaf JS-side instead of forcing an R round-trip. This was the long-deferred Phase 0c-C5 work item.
  3. A View Source panel that didn’t lie. The pre-0.2 JS tab in the in-widget source modal dumped the resolved WebSpec as a JSON literal — accurate, but a poor advertisement for the new authoring API. Targets: hide data behind a placeholder, emit theme: "lancet" when a preset matched, drop arguments at their defaults, use the new function builders.

Plus a bilingual docs example to prove the whole story end-to-end.

What landed in 0.2.0

The whole 0.2.0 round came in under a single working day of session time, ~4,500 lines added across 28 files. The pieces:

Cascade resolver port (srcjs/src/lib/theme-resolve.ts). Direct port of R/utils-theme-resolve.R’s resolve_theme() pipeline — Tier 1 mirror chain → chrome derivation → data slot bundles → text role derivation → density-driven spacing → Tier 3 cluster fills → construction-time contrast validation. The TS port came in at ~600 lines (vs the R original’s ~680) — slightly tighter because TS’s obj.field ?? defaults.field syntax is shorter than R’s fill_na(obj, list(...)) indirection. Already-existing TS-side OKLCH helpers in oklch.ts (Björn Ottosson’s transform from sRGB) got two new functions — contrastRatio (WCAG 2.1) and ensureContrast (walk L until the target ratio is met) — to back the validator.

22 column builders (srcjs/src/authoring/columns.ts). One per R col_*() helper: colText, colNumeric, colN, colInterval, colPvalue, colBar, colSparkline, colBadge, colIcon, colStars, colPictogram, colRing, colImg, colReference, colRange, colHeatmap, colProgress, colCurrency, colPercent, colEvents, colGroup, plus colLabel. Each takes a single options object (R-positional field argument becomes TS-named { field }); defaults match R verbatim where possible.

Viz builders + reference lines. vizForest, vizBar, vizBoxplot, vizViolin, plus effectForest / effectBar / effectBoxplot / effectViolin for multi-effect plots, and refline for annotations. Same pattern.

Theme API (srcjs/src/lib/theme-api.ts). webTheme(args) constructor, setInputs / setVariants / setSpacing / setThemeField modifiers, resolveThemeRef for the name-string-or-extend-object resolution path. Seven preset constructors covering the four journal themes (Cochrane, Lancet, JAMA, Dark) plus three LOTR easter-egg themes (Dwarven, Elvish, Hobbit) ported from R/themes-lotr.R.

Top-level tabviz() (srcjs/src/authoring/tabviz.ts). Takes a row array + column array + theme reference + labels/interaction toggles, returns a fully-formed WebSpec. Mounted with the existing createTabviz(node, spec) runtime factory.

View Source rewrite (srcjs/src/lib/source-emit.ts). Replaces the 200-line JSON dump with a pure emitJsSource(spec, opLog) function that walks the spec, infers which builder produced each column, drops arguments at their builder defaults, emits theme: "lancet" for preset matches, and hides the data array behind a tabvizData placeholder. Same op-log replay below the spec block.

Bilingual gallery example (docs/gallery/genetic-association.qmd). One Quarto page, two renders of the same forest plot — R chunk above, TypeScript reference below. The TS block was initially eval: false (reference only); we’d make it live in 0.2.1.

The OKLab precision discovery

The cascade-resolver port produced the resolved Tier 2/3 leaves, and we wanted to verify byte-equality against the R-resolved snapshot (theme-presets-v2.json was captured from tabviz:::serialize_theme( web_theme_cochrane()) and friends). The first test of cochrane against the snapshot:

chrome — surface (within OKLab precision)        PASS
chrome — content                                  PASS
chrome — divider                                  PASS
chrome — accent                                   FAIL
  fillEmphasis: TS "#0089B7" vs R "#008ACF"  (B channel differs by 24)
  stroke:       TS "#0079A2" vs R "#007AAC"  (10-channel drift)

The differences clustered on chroma-related derivations near gamut boundaries. Pure mixes (surface.muted, divider.subtle) matched exactly or within a single channel. Anything that ran through oklch_chroma() (slot bundle fillEmphasis / strokeEmphasis, accent muted/tints near the saturation edge) drifted by 5-25 channels per pixel.

After about an hour of investigation:

R> farver::convert_colour(farver::decode_colour("#0099CC"), "rgb", "oklch")
[1,]  0.6405375  0.1292621  231.0822

TS> hexToOklch("#0099CC")
{ L: 0.640533, C: 0.129199, H: 231.05493 }

R’s farver and our hand-rolled Ottosson transform produced OKLCH triples that differed by ~5e-6 L / ~6e-5 C / ~0.03° H. Negligible in isolation. But OKLab→sRGB gamut clipping bisects on chroma when the result is out-of-gamut, and tiny precision differences amplify dramatically when the result sits at the gamut boundary — which is exactly where a freshly chroma-boosted color tends to land.

Three paths considered: profile farver against Ottosson to find the coefficient delta and tune TS to match (cheapest if the delta turned out to be a single threshold); swap to a third-party OKLab implementation that matched farver (adds a dependency); or invert the parity direction — make TS canonical for the JSON snapshot, leave R’s runtime cascade alone, accept that each runtime produces internally consistent (but pairwise sub-perceptual-different) output.

For 0.2.0 we shipped option C in a relaxed form: tolerance-based parity test (per-channel delta ≤ 25), r-ts-parity-notes.md documenting the gap, and a flag that the canonization would be the next round’s work. Released. Moved on.

The 0.2.1 round: canonization + R streamlining

0.2.0 was Tuesday morning’s work. By Tuesday afternoon we were revisiting the parity gap with fresh eyes.

The canonization itself was four steps:

  1. Write srcjs/scripts/regenerate-theme-presets.ts — a Bun script that calls resolveTheme() on each of the seven preset Tier 1 input drafts and writes theme-presets-v2.json.
  2. Run it. The new snapshot looks structurally identical to the old one; individual hex values drift 1-25 channels on the chroma-related leaves.
  3. Tighten theme-resolve.test.ts from tolerance-based comparison to byte-exact toEqual — the parity test is now “does the resolver still produce the same shape it did when we last regenerated the snapshot.”
  4. Rebuild the JS bundle. The R-side resolver continues to run for R-rendered widgets; the snapshot is consumed only by the JS-side theme switcher and the View Source emitter’s preset-name matching.

The argument that each runtime is canonical in its own context fell out naturally: R-rendered widgets ship the wire with R-resolved values verbatim, so JS reads them as-is and renders without consulting any snapshot; JS-side programmatic authoring (tabviz({ theme: "lancet" }) and friends) and the in-widget theme switcher use the snapshot, so they get the TS values consistently. The two paths only meet in the specific scenario of “R-rendered widget, user clicks Lancet in the switcher” — and the difference there is sub-perceptual.

While porting the resolver to TS, we noticed three patterns in the R code that were paper-thin candidates for cleanup. They went into the 0.2.1 round along with the canonization:

Deduped legacy-input migration. web_theme() and set_inputs() both hand-rolled the same brand/tertiary v1→v2 deprecation block (the deprecated input field names that the v2 cascade rework removed). Extracted to a single check_legacy_inputs(args, arg_hint) helper in R/themes-api.R; the two callsites pass their respective migration-tip text. Saved ~18 lines.

Unified fill_na() and compose_text(). Both iterated S7 property names and wrote where the current value was NA. The difference: fill_na read defaults from a named list, compose_text from another S7 object. Collapsed into a single fill_na(obj, source) that dispatches on inherits(source, "S7_object"); the older compose_text(over, under) remains as a thin alias for callsite readability.

Dropped a defensive guard at the old line 583 of R/utils-theme-resolve.R. resolve_components had an if (is.na(theme@inputs@secondary_deep)) primary_deep else secondary_deep fallback — but resolve_components runs AFTER resolve_inputs_mirrors, which guarantees secondary_deep is non-NA. The guard was unreachable. Removed; full R test suite (~600 tests) verified no regression.

All three landed in commit 63af90e along with the canonization.

The bilingual docs page becoming actually bilingual

The genetic-association page that shipped with 0.2.0 had the TS block as eval: false text — reference, not rendering. The next session opened with the question: can Quarto actually run the TS side?

Four candidate paths surfaced:

  1. ojs chunks. Quarto’s native ObservableJS engine, runs JS in the browser at page load. Dynamic-import @tabviz/core from esm.sh; share data from R via ojs_define() + transpose(); mount via createTabviz(host, spec) into a host div the cell creates.
  2. Pre-render to SVG via @tabviz/core/export at site-build time.
  3. Standalone HTML file loaded via <iframe>.
  4. A custom Quarto Lua extension that natively understands tabviz-ts blocks.

Option 1 won on the integration vs. effort tradeoff. The first implementation came in about 30 lines of ojs + 5 lines of R-side ojs_define(). The first deploy attempt failed three times in sequence, each surfacing a real issue:

  • ojs_define() cannot be cached — fixed by #| cache: false on the R chunk.
  • TypeError: e.validate_each_keys is not a function — Svelte version drift. The 0.2.1 dist was compiled against svelte@5.46.1 (per srcjs/package-lock.json); esm.sh’s default resolution of ^5.0.0 pulled 5.55.8, whose runtime had reshuffled internal helpers. Fixed by pinning the version in the esm.sh import URL via ?deps=svelte@5.46.1.
  • A 502 from github.com/quarto-dev/quarto-cli/releases/download/... — unrelated infra hiccup. Re-ran the workflow.

After landing the fix we noticed the hardcoded 5.46.1 would break silently the next time anyone bumped Svelte in srcjs/package.json. Auto-pinned: the R chunk reads srcjs/package-lock.json::packages["node_modules/svelte"]$version and exposes it via ojs_define(svelte_version = ...); the ojs import template-literals it in. Future Svelte bumps now propagate to the docs on the next quarto render automatically.

A second bilingual gallery page (docs/gallery/bilingual-dashboard.qmd) followed the same pattern for the non-forest column family — bars / sparklines / badges / rings / stars — to demonstrate that the bilingual claim covers more than forest plots.

The hover-legibility class of bugs

While reviewing the new theme switcher behavior we noticed JAMA hovers were illegible — black-on-black on multiple menu items and toolbar buttons. The root cause: CSS rules using bare identity tokens (var(--tv-border), var(--tv-accent), var(--tv-primary-deep)) as hover backgrounds. Fine when those tokens are mid-tone, broken when a theme pins them dark — JAMA’s accent: #000000 and divider.subtle: #000000, dark theme’s primary_deep: #2E5290.

The pre-0.2 codebase had already established the right pattern in some places via color-mix(in srgb, var(--tv-accent) 12%, var(--tv-bg)) for hover backgrounds (a 12% accent tint that’s guaranteed legible because it tints into the surface). The fix: codify the convention, introduce a single --tv-hover-bg derived variable at the ForestPlot.svelte variable declaration block, audit all :hover / :active / .open rules for bare-identity-token backgrounds, and migrate them. Result: 13 sites migrated across ZoomControls, ThemeSwitcher, TokenPicker, five button components, three controls components, and two popovers. The decorative-divider uses of --tv-border (1px lines, bar-track backgrounds) stayed put — those don’t carry text on top.

The R→TS delegation pilot

The original Phase 2 plan was R helpers delegating construction to the TS authoring builders via V8 — the “single source of truth” move. The initial sketch had a V8-when-available pattern: delegate if V8 was loaded (Suggests), fall back to the hand-rolled R path otherwise.

The pushback from the user — “let’s not maintain parallel stacks” — landed quickly. Three changes:

  1. V8 promoted from Suggests → Imports. DESCRIPTION updated. V8 is now a hard runtime dep; the R package always has access to the JS runtime.
  2. R/v8-bridge.R — a singleton V8 context, lazy-initialized, plus a ts_call(builder_name, args) helper that dispatches via inst/js/svg-generator.js’s new callBuilder(name, argsJson) global (added to srcjs/src/export/v8-entry.ts so the same bundle covers SVG export + authoring delegation).
  3. col_text() refactored to delegate. The R-side helper now pushes argument validation through checkmate, then calls ts_call("colText", ts_args) to compute the wire shape (id, header default, options bundle), then wraps the result in an S7 ColumnSpec via web_col() for R-side style mappings + formatter slot compatibility.

The do.call boilerplate inside col_text was a small let-the-caller-override dance: when tabviz() internally calls col_text(field = "study", id = "label") to construct the row-label column, the explicit id = "label" arg arrives via ... and needs to shadow the TS-computed default. do.call(web_col, list(...) <- overrides) was cleaner than parameter juggling.

Only col_text got the treatment in this round — pilot rather than fleet refactor. The pattern is now established; the other 21 column builders + 4 viz builders can follow when there’s value.

The R↔︎TS column parity test harness

The other 21 helpers staying R-only is fine because the parity harness now catches drift mechanically. tests/testthat/test-parity-columns.R exercises each column helper R-side and TS-side with the same args via ts_call(...), then asserts key wire-shape fields (id, field, header, type, options structure) match.

It caught three real drifts in its first run:

col_stars vs colStars. R implements stars as a thin wrapper over col_pictogram(glyph = "star") — wire type is “pictogram”, not “stars”. TS colStars was building its own “stars”-typed column. Aligned TS by making colStars defer to colPictogram.

col_interval field naming. R uses synthetic _interval_<point> field names so two interval columns sharing the same point estimate get distinct ids. TS was using the bare point as the field. Mirrored the synthetic prefix convention.

col_sparkline default header. R defaults header = "Trend" (literal); TS had it falling through to the field name. Mirrored.

Each was a one-line fix on the TS side. The drift detection paid for itself within ten minutes of writing the harness.

Spec modifiers (srcjs/src/authoring/modifiers.ts)

The R API has fluent set_* modifiers that work on either a static WebSpec or a Shiny proxy: set_title(), set_subtitle(), set_caption(), set_footnote(), set_theme(), set_zoom(), add_column(), remove_column(), update_column(), plus paint_row(), paint_cell(), sort_rows(), filter_rows().

The static-spec subset got TS mirrors in 0.2.1’s follow-up round — pure functions, immutable, composable. Runtime control of a mounted widget continues to use the instance methods returned by createTabviz (instance.sortBy(...) etc); the modifier API here is for spec authoring, not live updates. The Shiny proxy variant stays R-only — purely an R-concern.

What didn’t ship

The 0.2.x line’s “out of scope” list got smaller over the round but isn’t empty:

  • Full R→TS delegation across all builders. The col_text pilot validated the pattern; the remaining 21 column helpers + 4 viz helpers + theme constructors still hand-roll their wire shapes R-side. The parity tests catch drift; full delegation can come when there’s concrete value or when the maintenance burden of two implementations starts to bite.
  • Shiny proxy in TS. The R-side proxy API (tabviz_proxy(session, output_id) |> set_title(...) |> ...) is fundamentally an R-concern (Shiny is an R runtime); there’s no obvious analog in pure TS. Not on the radar.
  • Bilingual conversion of the older gallery pages. Two new bilingual pages got added (genetic-association + bilingual-dashboard). The existing meta-analysis / clinical / dashboards / viz-glyphs pages stay R-only for now — retrofitting them is real time investment with diminishing returns past the first couple of examples.

Numbers

  • Commits: 4 on main between 0cc9f75 (the 0.31.1 tag) and whatever lands after this devblog.
  • Lines added: ~6,500 across the two rounds (4,500 in 0.2.0, ~2,000 in 0.2.1’s follow-up).
  • Test suite: 252 → 262 TS tests + the full R suite + the new R parity test file. All green.
  • Bundle size: main npm entry 5.3 KB gzipped (≈ unchanged from 0.1.5 after the authoring layer landed; the cascade resolver + presets-inputs added ~22 KB to the IIFE which is within the bumped size budget).
  • Tagged: v0.31.2 on main, @tabviz/core@0.2.0 and @0.2.1 on the npm registry.

Closing notes

The 0.2.x line was conceived as “make the JS side a first-class authoring target.” That happened. The OKLab gap discovery and its eventual resolution were the most interesting detour: it surfaced an architectural choice that hadn’t been forced before (which runtime owns the canonical theme snapshot?), and the answer that fell out — each runtime canonical in its own context — generalizes to other R↔︎TS divergences that might surface later. The bilingual gallery pages exist as proof, not just claim. And the hover-legibility cleanup was a pure quality-of-life win that came along for free because the audit framework (grep for bare identity tokens, migrate to a single new variable) was straightforward once the convention was named.

Next obvious extensions: more col_* helpers delegated through V8 if the maintenance burden warrants it; more bilingual gallery pages if authors want them; potentially a custom Quarto extension that natively understands tabviz-ts blocks so future bilingual pages don’t need the ojs + esm.sh + svelte-version-pin scaffolding by hand. But none of those are urgent. The 0.2.x line is shipped.

```sh
TS> hexToOklch("#0099CC")
{ L: 0.640533, C: 0.129199, H: 231.05493 }
```

R's `farver` and our hand-rolled Ottosson transform produced OKLCH
triples that differed by ~5e-6 L / ~6e-5 C / ~0.03° H. Negligible in
isolation. But OKLab→sRGB gamut clipping bisects on chroma when the
result is out-of-gamut, and tiny precision differences amplify
dramatically when the result sits *at* the gamut boundary — which is
exactly where a freshly chroma-boosted color tends to land.

Three paths considered: profile farver against Ottosson to find the
coefficient delta and tune TS to match (cheapest if the delta turned
out to be a single threshold); swap to a third-party OKLab
implementation that matched farver (adds a dependency); or invert the
parity direction — make TS canonical for the JSON snapshot, leave R's
runtime cascade alone, accept that each runtime produces internally
consistent (but pairwise sub-perceptual-different) output.

For 0.2.0 we shipped option C in a relaxed form: tolerance-based
parity test (per-channel delta ≤ 25), `r-ts-parity-notes.md`
documenting the gap, and a flag that the canonization would be the
next round's work. Released. Moved on.

## The 0.2.1 round: canonization + R streamlining

0.2.0 was Tuesday morning's work. By Tuesday afternoon we were
revisiting the parity gap with fresh eyes.

The canonization itself was four steps:

1. Write `srcjs/scripts/regenerate-theme-presets.ts` — a Bun script
   that calls `resolveTheme()` on each of the seven preset Tier 1
   input drafts and writes `theme-presets-v2.json`.
2. Run it. The new snapshot looks structurally identical to the old
   one; individual hex values drift 1-25 channels on the
   chroma-related leaves.
3. Tighten `theme-resolve.test.ts` from tolerance-based comparison
   to byte-exact `toEqual` — the parity test is now "does the
   resolver still produce the same shape it did when we last
   regenerated the snapshot."
4. Rebuild the JS bundle. The R-side resolver continues to run for
   R-rendered widgets; the snapshot is consumed only by the JS-side
   theme switcher and the View Source emitter's preset-name matching.

The argument that *each runtime is canonical in its own context* fell
out naturally: R-rendered widgets ship the wire with R-resolved values
verbatim, so JS reads them as-is and renders without consulting any
snapshot; JS-side programmatic authoring (`tabviz({ theme: "lancet"
})` and friends) and the in-widget theme switcher use the snapshot, so
they get the TS values consistently. The two paths only meet in the
specific scenario of "R-rendered widget, user clicks Lancet in the
switcher" — and the difference there is sub-perceptual.

While porting the resolver to TS, we noticed three patterns in the R
code that were paper-thin candidates for cleanup. They went into the
0.2.1 round along with the canonization:

**Deduped legacy-input migration.** `web_theme()` and `set_inputs()`
both hand-rolled the same `brand`/`tertiary` v1→v2 deprecation block
(the deprecated input field names that the v2 cascade rework
removed). Extracted to a single `check_legacy_inputs(args, arg_hint)`
helper in `R/themes-api.R`; the two callsites pass their respective
migration-tip text. Saved ~18 lines.

**Unified `fill_na()` and `compose_text()`.** Both iterated S7
property names and wrote where the current value was `NA`. The
difference: `fill_na` read defaults from a named list,
`compose_text` from another S7 object. Collapsed into a single
`fill_na(obj, source)` that dispatches on
`inherits(source, "S7_object")`; the older `compose_text(over, under)`
remains as a thin alias for callsite readability.

**Dropped a defensive guard at the old line 583** of
`R/utils-theme-resolve.R`. `resolve_components` had an
`if (is.na(theme@inputs@secondary_deep)) primary_deep else
secondary_deep` fallback — but `resolve_components` runs AFTER
`resolve_inputs_mirrors`, which guarantees `secondary_deep` is
non-NA. The guard was unreachable. Removed; full R test suite (~600
tests) verified no regression.

All three landed in commit `63af90e` along with the canonization.

## The bilingual docs page becoming actually bilingual

The genetic-association page that shipped with 0.2.0 had the TS
block as `eval: false` text — reference, not rendering. The next
session opened with the question: can Quarto actually run the TS
side?

Four candidate paths surfaced:

1. **`ojs` chunks.** Quarto's native ObservableJS engine, runs JS in
   the browser at page load. Dynamic-import `@tabviz/core` from
   esm.sh; share data from R via `ojs_define()` + `transpose()`;
   mount via `createTabviz(host, spec)` into a host div the cell
   creates.
2. Pre-render to SVG via `@tabviz/core/export` at site-build time.
3. Standalone HTML file loaded via `<iframe>`.
4. A custom Quarto Lua extension that natively understands
   `tabviz-ts` blocks.

Option 1 won on the integration vs. effort tradeoff. The first
implementation came in about 30 lines of ojs + 5 lines of R-side
`ojs_define()`. The first deploy attempt failed three times in
sequence, each surfacing a real issue:

- `ojs_define() cannot be cached` — fixed by `#| cache: false` on the
  R chunk.
- `TypeError: e.validate_each_keys is not a function` — Svelte
  version drift. The 0.2.1 dist was compiled against
  `svelte@5.46.1` (per `srcjs/package-lock.json`); esm.sh's default
  resolution of `^5.0.0` pulled `5.55.8`, whose runtime had
  reshuffled internal helpers. Fixed by pinning the version in the
  esm.sh import URL via `?deps=svelte@5.46.1`.
- A 502 from `github.com/quarto-dev/quarto-cli/releases/download/...`
  — unrelated infra hiccup. Re-ran the workflow.

After landing the fix we noticed the hardcoded `5.46.1` would break
silently the next time anyone bumped Svelte in `srcjs/package.json`.
Auto-pinned: the R chunk reads
`srcjs/package-lock.json::packages["node_modules/svelte"]$version`
and exposes it via `ojs_define(svelte_version = ...)`; the ojs
import template-literals it in. Future Svelte bumps now propagate
to the docs on the next quarto render automatically.

A second bilingual gallery page
(`docs/gallery/bilingual-dashboard.qmd`) followed the same pattern
for the *non-forest* column family — bars / sparklines / badges /
rings / stars — to demonstrate that the bilingual claim covers more
than forest plots.

## The hover-legibility class of bugs

While reviewing the new theme switcher behavior we noticed JAMA
hovers were illegible — black-on-black on multiple menu items and
toolbar buttons. The root cause: CSS rules using bare identity
tokens (`var(--tv-border)`, `var(--tv-accent)`,
`var(--tv-primary-deep)`) as hover backgrounds. Fine when those
tokens are mid-tone, broken when a theme pins them dark — JAMA's
`accent: #000000` and `divider.subtle: #000000`, dark theme's
`primary_deep: #2E5290`.

The pre-0.2 codebase had already established the right pattern in
some places via `color-mix(in srgb, var(--tv-accent) 12%, var(--tv-bg))`
for hover backgrounds (a 12% accent tint that's guaranteed legible
because it tints into the surface). The fix: codify the convention,
introduce a single `--tv-hover-bg` derived variable at the
`ForestPlot.svelte` variable declaration block, audit all `:hover` /
`:active` / `.open` rules for bare-identity-token backgrounds, and
migrate them. Result: 13 sites migrated across `ZoomControls`,
`ThemeSwitcher`, `TokenPicker`, five button components, three
controls components, and two popovers. The decorative-divider uses
of `--tv-border` (1px lines, bar-track backgrounds) stayed put —
those don't carry text on top.

## The R→TS delegation pilot

The original Phase 2 plan was R helpers delegating construction to
the TS authoring builders via V8 — the "single source of truth"
move. The initial sketch had a V8-when-available pattern: delegate
if `V8` was loaded (Suggests), fall back to the hand-rolled R path
otherwise.

The pushback from the user — "let's not maintain parallel stacks" —
landed quickly. Three changes:

1. **V8 promoted from Suggests → Imports.** `DESCRIPTION` updated.
   V8 is now a hard runtime dep; the R package always has access
   to the JS runtime.
2. **`R/v8-bridge.R`** — a singleton V8 context, lazy-initialized,
   plus a `ts_call(builder_name, args)` helper that dispatches via
   `inst/js/svg-generator.js`'s new `callBuilder(name, argsJson)`
   global (added to `srcjs/src/export/v8-entry.ts` so the same
   bundle covers SVG export + authoring delegation).
3. **`col_text()` refactored to delegate.** The R-side helper now
   pushes argument validation through `checkmate`, then calls
   `ts_call("colText", ts_args)` to compute the wire shape (id,
   header default, options bundle), then wraps the result in an
   S7 `ColumnSpec` via `web_col()` for R-side style mappings +
   formatter slot compatibility.

The `do.call` boilerplate inside `col_text` was a small
let-the-caller-override dance: when `tabviz()` internally calls
`col_text(field = "study", id = "label")` to construct the row-label
column, the explicit `id = "label"` arg arrives via `...` and needs
to shadow the TS-computed default. `do.call(web_col, list(...) <-
overrides)` was cleaner than parameter juggling.

Only `col_text` got the treatment in this round — pilot rather than
fleet refactor. The pattern is now established; the other 21 column
builders + 4 viz builders can follow when there's value.

## The R↔TS column parity test harness

The other 21 helpers staying R-only is fine *because* the parity
harness now catches drift mechanically. `tests/testthat/test-parity-columns.R`
exercises each column helper R-side and TS-side with the same args
via `ts_call(...)`, then asserts key wire-shape fields (id, field,
header, type, options structure) match.

It caught three real drifts in its first run:

**`col_stars` vs `colStars`.** R implements stars as a thin wrapper
over `col_pictogram(glyph = "star")` — wire type is "pictogram", not
"stars". TS `colStars` was building its own "stars"-typed column.
Aligned TS by making `colStars` defer to `colPictogram`.

**`col_interval` field naming.** R uses synthetic `_interval_<point>`
field names so two interval columns sharing the same point estimate
get distinct ids. TS was using the bare `point` as the field. Mirrored
the synthetic prefix convention.

**`col_sparkline` default header.** R defaults `header = "Trend"`
(literal); TS had it falling through to the field name. Mirrored.

Each was a one-line fix on the TS side. The drift detection paid
for itself within ten minutes of writing the harness.

## Spec modifiers (`srcjs/src/authoring/modifiers.ts`)

The R API has fluent `set_*` modifiers that work on either a static
`WebSpec` or a Shiny proxy: `set_title()`, `set_subtitle()`,
`set_caption()`, `set_footnote()`, `set_theme()`, `set_zoom()`,
`add_column()`, `remove_column()`, `update_column()`, plus
`paint_row()`, `paint_cell()`, `sort_rows()`, `filter_rows()`.

The static-spec subset got TS mirrors in 0.2.1's follow-up round —
pure functions, immutable, composable. Runtime control of a mounted
widget continues to use the instance methods returned by
`createTabviz` (`instance.sortBy(...)` etc); the modifier API here
is for spec authoring, not live updates. The Shiny proxy variant
stays R-only — purely an R-concern.

## What didn't ship

The 0.2.x line's "out of scope" list got smaller over the round but
isn't empty:

- **Full R→TS delegation across all builders.** The `col_text` pilot
  validated the pattern; the remaining 21 column helpers + 4 viz
  helpers + theme constructors still hand-roll their wire shapes
  R-side. The parity tests catch drift; full delegation can come
  when there's concrete value or when the maintenance burden of two
  implementations starts to bite.
- **Shiny proxy in TS.** The R-side proxy API
  (`tabviz_proxy(session, output_id) |> set_title(...) |> ...`) is
  fundamentally an R-concern (Shiny is an R runtime); there's no
  obvious analog in pure TS. Not on the radar.
- **Bilingual conversion of the older gallery pages.** Two new
  bilingual pages got added (genetic-association +
  bilingual-dashboard). The existing meta-analysis / clinical /
  dashboards / viz-glyphs pages stay R-only for now — retrofitting
  them is real time investment with diminishing returns past the
  first couple of examples.

## Closing notes

The 0.2.x line was conceived as "make the JS side a first-class
authoring target." That happened. The OKLab gap discovery and its
eventual resolution were the most interesting detour: it surfaced an
architectural choice that hadn't been forced before (which runtime
owns the canonical theme snapshot?), and the answer that fell out —
each runtime canonical in its own context — generalizes to other
R↔TS divergences that might surface later. The bilingual gallery
pages exist as proof, not just claim. And the hover-legibility cleanup
was a pure quality-of-life win that came along for free because the
audit framework (grep for bare identity tokens, migrate to a single
new variable) was straightforward once the convention was named.

Next obvious extensions: more `col_*` helpers delegated through V8 if
the maintenance burden warrants it; more bilingual gallery pages if
authors want them; potentially a custom Quarto extension that
natively understands `tabviz-ts` blocks so future bilingual pages
don't need the ojs + esm.sh + svelte-version-pin scaffolding by hand.
But none of those are urgent. The 0.2.x line is shipped.
