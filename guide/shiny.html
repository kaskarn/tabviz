Shiny Integration

tabviz works seamlessly with Shiny through the htmlwidgets framework. This page covers basic integration, the proxy API for live updates, and split-table outputs.

Output and Render

UI Server Used for
tabvizOutput() renderTabviz() A single tabviz widget
splitTabvizOutput() renderSplitTabviz() A SplitForest (tabviz + sidebar of subgroups) 
library(shiny)
library(tabviz)

ui <- fluidPage(
  titlePanel("Forest Plot Explorer"),
  tabvizOutput("plot", height = "500px")
)

server <- function(input, output, session) {
  output$plot <- renderTabviz({
    tabviz(meta_data,
      label = "study",
      columns = list(
        viz_forest(point = "hr", lower = "lower", upper = "upper",
                   scale = "log", null_value = 1)
      )
    )
  })
}

shinyApp(ui, server)

Sizing

Both output functions accept standard CSS sizing:

Argument Default Examples
width "100%" "800px", "50vw"
height "400px" ("600px" for split) "600px", "80vh"

Reactive Data Patterns

Connect the plot to reactive sources — the widget re-renders whenever the reactive changes:

server <- function(input, output, session) {
  filtered_data <- reactive({
    meta_data |>
      dplyr::filter(region == input$region)
  })

  output$plot <- renderTabviz({
    tabviz(filtered_data(),
      label = "study",
      columns = list(
        viz_forest(point = "hr", lower = "lower", upper = "upper",
                   scale = "log", null_value = 1)
      )
    )
  })
}

For inputs that change frequently (sliders), debounce to avoid excessive re-renders:

filter_value <- reactive(input$min_n) |>
  shiny::debounce(300)

The Proxy API

Proxy functions update the plot without a full re-render — scroll position, selection state, and layout stay stable.

Function Purpose
tabviz_proxy(id) Reference a tabvizOutput for live updates
split_tabviz_proxy(id) Reference a splitTabvizOutput for live updates
split_tabviz_select(proxy, key) Switch a split table to a specific subgroup
ImportantAlways create proxies inside reactive contexts

Never create a proxy at the top level of your app. Both tabviz_proxy() and split_tabviz_proxy() require an active Shiny reactive domain.

Fluent verbs dispatch on proxies

Every fluent modifier — every set_*, add_*, remove_*, sort_*, filter_*, plus set_cell() and clear_edits() — dispatches on tabviz_proxy and sends a custom message to the running widget instead of mutating a static spec. The same call site works on a WebSpec, an htmlwidget, and a proxy:

Category Verbs
Columns add_column(), remove_column(), move_column(), resize_column(), update_column()
Rows sort_rows(), filter_rows(), clear_filters(), select_rows(), move_row()
Data update_data() (proxy requires a fresh WebSpec)
Groups toggle_group()
Cells set_cell(), set_row_label(), clear_edits() (proxy-only)
Global set_theme(), set_zoom()

Sorting and filtering

observeEvent(input$sort_asc,  sort_rows(tabviz_proxy("plot"), "hr", "asc"))
observeEvent(input$sort_desc, sort_rows(tabviz_proxy("plot"), "hr", "desc"))

observeEvent(input$filter_n, {
  filter_rows(tabviz_proxy("plot"), "n", "gt", input$min_sample_size)
})

observeEvent(input$clear_filters, {
  clear_filters(tabviz_proxy("plot"))
})

Filter operators: "eq", "neq", "gt", "lt", "contains".

Replacing the dataset

update_data() on a proxy takes a WebSpec, not a raw data frame. Build one with tabviz(..., .spec_only = TRUE):

observeEvent(input$switch_dataset, {
  new_spec <- tabviz(
    get_dataset(input$dataset_name),
    label = "study",
    columns = list(
      viz_forest(point = "hr", lower = "lower", upper = "upper",
                 scale = "log", null_value = 1)
    ),
    .spec_only = TRUE
  )
  update_data(tabviz_proxy("plot"), new_spec)
})

Chaining

Every proxy verb returns the proxy invisibly, so calls chain:

observeEvent(input$reset_view, {
  tabviz_proxy("plot") |>
    clear_filters() |>
    sort_rows("study", "none") |>
    toggle_group("all", collapsed = FALSE)
})

Split Tables in Shiny

Use splitTabvizOutput() / renderSplitTabviz() for a SplitForest. Programmatically focus a subgroup with split_tabviz_select():

ui <- fluidPage(
  selectInput("region", "Region",
    choices = c("North America", "Europe", "Asia Pacific")),
  splitTabvizOutput("split_plot", height = "700px")
)

server <- function(input, output, session) {
  output$split_plot <- renderSplitTabviz({
    tabviz(trial_data,
      label = "study",
      columns = list(
        viz_forest(point = "hr", lower = "lower", upper = "upper",
                   scale = "log", null_value = 1)
      )
    ) |>
      split_table(by = "region")
  })

  observeEvent(input$region, {
    split_tabviz_select(split_tabviz_proxy("split_plot"), input$region)
  })
}

Complete Example App

library(shiny)
library(tabviz)
library(dplyr)

meta_data <- data.frame(
  study = c("ADVANCE", "SPRINT", "ACCORD", "ONTARGET", "HOPE-3",
            "EMPA-REG", "CANVAS", "DECLARE", "DAPA-HF", "EMPEROR"),
  drug_class = c("ACEi", "Intensive BP", "Intensive BP", "ARB", "Statin",
                 "SGLT2i", "SGLT2i", "SGLT2i", "SGLT2i", "SGLT2i"),
  hr    = c(0.91, 0.75, 0.88, 0.94, 0.90, 0.62, 0.86, 0.93, 0.74, 0.75),
  lower = c(0.83, 0.64, 0.76, 0.86, 0.82, 0.49, 0.75, 0.84, 0.65, 0.65),
  upper = c(1.01, 0.87, 1.01, 1.02, 0.98, 0.77, 0.97, 1.03, 0.85, 0.86),
  n     = c(11140, 9361, 10251, 25620, 12705, 7020, 10142, 17160, 4744, 3730)
)

ui <- fluidPage(
  titlePanel("Interactive Forest Plot"),
  sidebarLayout(
    sidebarPanel(
      width = 3,
      h4("Filters"),
      selectInput("drug_class", "Drug Class",
        choices = c("All", unique(meta_data$drug_class)), selected = "All"),
      sliderInput("min_n", "Minimum Sample Size",
        min = 0, max = 20000, value = 0, step = 1000),
      hr(),
      h4("Display"),
      selectInput("theme", "Theme",
        choices = c("Default" = "default", "JAMA" = "jama", "Lancet" = "lancet")),
      actionButton("sort_hr", "Sort by HR"),
      actionButton("reset",   "Reset View")
    ),
    mainPanel(width = 9, tabvizOutput("plot", height = "600px"))
  )
)

server <- function(input, output, session) {
  plot_data <- reactive({
    d <- meta_data
    if (input$drug_class != "All") d <- dplyr::filter(d, drug_class == input$drug_class)
    dplyr::filter(d, n >= input$min_n)
  })

  output$plot <- renderTabviz({
    tabviz(plot_data(),
      label = "study",
      group = "drug_class",
      columns = list(
        col_numeric("n", "N"),
        viz_forest(point = "hr", lower = "lower", upper = "upper",
                   scale = "log", null_value = 1, axis_label = "Hazard Ratio")
      )
    )
  })

  observe({
    set_theme(tabviz_proxy("plot"), switch(input$theme,
      "jama" = "jama", "lancet" = "lancet", "default"))
  })

  sort_state <- reactiveVal("none")
  observeEvent(input$sort_hr, {
    new_dir <- switch(sort_state(), "none" = "asc", "asc" = "desc", "desc" = "none")
    sort_state(new_dir)
    sort_rows(tabviz_proxy("plot"), "hr", new_dir)
  })

  observeEvent(input$reset, {
    updateSelectInput(session, "drug_class", selected = "All")
    updateSliderInput(session, "min_n", value = 0)
    sort_state("none")
    tabviz_proxy("plot") |>
      clear_filters() |>
      sort_rows("hr", "none")
  })
}

shinyApp(ui, server)

Two-way Integration: Reading Widget State

The widget emits every interactive dimension back to Shiny as named inputs. This lets a dashboard mirror what the user is doing inside the widget — sort clicks, filter changes, column show/hide, paint tokens, group collapses, even zoom and pan state.

The contract

Every dimension is emitted as input$<id>_<dimension> with a uniform envelope:

list(value = ..., source = "user" | "proxy", ts = <ms>)

source distinguishes user-driven mutations from proxy-pushed ones, so an observer can ignore its own writes:

observeEvent(input$plot_sort, {
  req(input$plot_sort$source == "user")  # don't echo proxy writes
  current_sort <- input$plot_sort$value  # list(column =, direction =)
  # ...react to user-driven sort change
})

tabviz_state() — read everything at once

tabviz_state(input, "plot") pulls every dimension into a single named list, unwrapped. Use tabviz_state_envelope() if you need the source and ts.

observe({
  state <- tabviz_state(input, "plot")
  str(state$sort)              # list(column = "hr", direction = "desc")
  str(state$filters)           # list(arm = list(operator = "eq", value = "A"))
  str(state$selected)          # c("study_1", "study_4")
  str(state$hidden_columns)    # c("se", "ci_lo")
  str(state$collapsed_groups)  # c("group_drug_class")
})

Dimensions emitted

Dimension Shape
sort list(column, direction) or NULL
filters named list, key = field, value = list(operator, value)
selected character() of row ids
hover row id or NULL
collapsed_groups character() of group ids
hidden_columns character() of column ids
column_order character() of column ids in display order
column_widths named numeric
cell_edits list(cells, groups) map of user-typed values
label_edits named list (title/subtitle/caption/footnote)
row_styles, cell_styles paint-token maps
paint_tool list(token, scope) — the active brush
zoom list(zoom, autoFit, maxWidth, maxHeight, showZoomControls)
axis_zooms per-forest-column pan/zoom domains
banding list(mode, startsWithBand) or NULL
plot_width numeric or NULL
visible_rows character() of row ids actually visible after filter+sort+collapse

There is also input$<id>_state, a debounced (~150 ms) bundle of every dimension above. Useful for bookmarking or debugging — observe(str(input$plot_state)) shows the whole world.

Round-trip example

ui <- fluidPage(
  selectInput("col", "Sort column", c("hr", "n")),
  selectInput("dir", "Direction", c("asc", "desc", "none")),
  tabvizOutput("plot"),
  verbatimTextOutput("status")
)

server <- function(input, output, session) {
  output$plot <- renderTabviz({
    tabviz(meta_data, label = "study",
      columns = list(viz_forest("hr", "lower", "upper")))
  })

  # Shiny → Widget: dropdowns drive the widget's sort.
  observe({
    sort_rows(tabviz_proxy("plot"), input$col, input$dir)
  })

  # Widget → Shiny: clicking a header in the widget updates the dropdowns.
  observeEvent(input$plot_sort, {
    req(input$plot_sort$source == "user")  # ignore our own pushes
    s <- input$plot_sort$value
    updateSelectInput(session, "col", selected = s$column)
    updateSelectInput(session, "dir", selected = s$direction)
  })

  output$status <- renderPrint(tabviz_state(input, "plot"))
}

The source == "user" guard is the key — without it, the proxy’s own writes would echo back into the dropdowns and you’d risk a feedback loop.

Initial State at First Paint

tabviz() accepts three initial_* args so the first paint already reflects the dashboard’s expected state — no flash of unsorted/unfiltered content before a proxy push lands:

output$plot <- renderTabviz({
  tabviz(meta_data, label = "study",
    initial_sort = list(column = "hr", direction = "desc"),
    initial_filters = list(arm = list(operator = "eq", value = "A")),
    initial_hidden_columns = c("se", "ci_lo"),
    columns = list(viz_forest("hr", "lower", "upper"))
  )
})

These are the constructor counterparts of sort_rows(), filter_rows(), and remove_column() — same shape, applied before mount.

Proxy vs Re-render

Use case Approach Reason
Sort / filter existing rows Proxy Smooth, no flicker
Change theme / zoom Proxy set_theme() / set_zoom() route through the store
Add, remove, resize columns Proxy Column verbs are wired through the store
Completely different dataset (new columns/types) Re-render or update_data() with a fresh WebSpec
Toggle groups Proxy UI-state only

Best Practices

  1. Create proxies inside reactive contexts, not at app top level.
  2. Debounce slider-driven inputs with shiny::debounce().
  3. Prefer proxy verbs for interactive updates — they preserve scroll and selection.
  4. Reach for full re-render only when the column structure or data schema changes.
  5. Chain proxy calls for atomic UI updates.

| `cell_edits` | `list(cells, groups)` map of user-typed values |
| `label_edits` | named list (title/subtitle/caption/footnote) |
| `row_styles`, `cell_styles` | paint-token maps |
| `paint_tool` | `list(token, scope)` — the active brush |
| `zoom` | `list(zoom, autoFit, maxWidth, maxHeight, showZoomControls)` |
| `axis_zooms` | per-forest-column pan/zoom domains |
| `banding` | `list(mode, startsWithBand)` or `NULL` |
| `plot_width` | numeric or `NULL` |
| `visible_rows` | `character()` of row ids actually visible after filter+sort+collapse |

There is also `input$<id>_state`, a debounced (~150 ms) bundle of every
dimension above. Useful for bookmarking or debugging — `observe(str(input$plot_state))`
shows the whole world.

### Round-trip example

```r
ui <- fluidPage(
  selectInput("col", "Sort column", c("hr", "n")),
  selectInput("dir", "Direction", c("asc", "desc", "none")),
  tabvizOutput("plot"),
  verbatimTextOutput("status")
)

server <- function(input, output, session) {
  output$plot <- renderTabviz({
    tabviz(meta_data, label = "study",
      columns = list(viz_forest("hr", "lower", "upper")))
  })

  # Shiny → Widget: dropdowns drive the widget's sort.
  observe({
    sort_rows(tabviz_proxy("plot"), input$col, input$dir)
  })

  # Widget → Shiny: clicking a header in the widget updates the dropdowns.
  observeEvent(input$plot_sort, {
    req(input$plot_sort$source == "user")  # ignore our own pushes
    s <- input$plot_sort$value
    updateSelectInput(session, "col", selected = s$column)
    updateSelectInput(session, "dir", selected = s$direction)
  })

  output$status <- renderPrint(tabviz_state(input, "plot"))
}
```

The `source == "user"` guard is the key — without it, the proxy's own writes
would echo back into the dropdowns and you'd risk a feedback loop.

## Initial State at First Paint

`tabviz()` accepts three `initial_*` args so the first paint already reflects
the dashboard's expected state — no flash of unsorted/unfiltered content
before a proxy push lands:

```r
output$plot <- renderTabviz({
  tabviz(meta_data, label = "study",
    initial_sort = list(column = "hr", direction = "desc"),
    initial_filters = list(arm = list(operator = "eq", value = "A")),
    initial_hidden_columns = c("se", "ci_lo"),
    columns = list(viz_forest("hr", "lower", "upper"))
  )
})
```

These are the constructor counterparts of `sort_rows()`, `filter_rows()`, and
`remove_column()` — same shape, applied before mount.

## Proxy vs Re-render

| Use case | Approach | Reason |
|---|---|---|
| Sort / filter existing rows | **Proxy** | Smooth, no flicker |
| Change theme / zoom | **Proxy** | `set_theme()` / `set_zoom()` route through the store |
| Add, remove, resize columns | **Proxy** | Column verbs are wired through the store |
| Completely different dataset (new columns/types) | **Re-render** or `update_data()` with a fresh `WebSpec` |
| Toggle groups | **Proxy** | UI-state only |

## Best Practices

1. Create proxies inside reactive contexts, not at app top level.
2. Debounce slider-driven inputs with `shiny::debounce()`.
3. Prefer proxy verbs for interactive updates — they preserve scroll and selection.
4. Reach for full re-render only when the column structure or data schema changes.
5. Chain proxy calls for atomic UI updates.
