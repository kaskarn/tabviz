# tabviz performance benchmarks

Synthetic benchmarks for the algorithmic hot paths. Designed to run in <1 minute
in Node via Bun — no browser, no R. Measures the parts of the pipeline that
scale with row/column/subset count: width measurement, theme resolution, CSS
emission, style extraction.

## Run

```bash
cd srcjs
bun run tests/perf/run-bench.ts
```

To save a baseline before a refactor:

```bash
bun run tests/perf/run-bench.ts > tests/perf/baseline.txt
```

After the refactor, diff:

```bash
bun run tests/perf/run-bench.ts > tests/perf/after.txt
diff tests/perf/baseline.txt tests/perf/after.txt
```

## Fixtures

Generated by `fixtures.ts`. Naming convention: `<size>_<shape>`.

| Fixture          | Rows × Cols | Notes                                      |
|------------------|-------------|--------------------------------------------|
| `small_mixed`    | 100 × 5     | Smoke; should always be <1ms / scenario.   |
| `medium_mixed`   | 1k × 10     | Typical "interesting" workload.            |
| `large_mixed`    | 5k × 10     | Stress upper bound of the target scale.    |
| `xl_mixed`       | 10k × 15    | Above target scale; tracks degradation.    |
| `split_small`    | 10 × 100    | Few small subsets.                         |
| `split_medium`   | 50 × 100    | The "many small splits" target workload.   |
| `split_large`    | 5 × 1000    | Few medium subsets.                        |

## What we measure

Each measurement is a synthetic stand-in for a real call site. They share
fixtures across runs to keep variance low. Reported as median of N samples.

| Scenario                  | What it stands in for                                  |
|---------------------------|--------------------------------------------------------|
| `measure.estimator`       | V8 path (`estimateTextWidth`) over all cells          |
| `measure.measureString`   | New unified API in unregistered mode (estimator)      |
| `measure.measureString*`  | Same API after `registerFontMetrics` (lookup)         |
| `theme.resolve`           | Cold theme cascade (`resolveTheme(draft)`)            |
| `theme.cssBuild`          | First call to `buildThemeCSS(theme)`                  |
| `theme.cssBuild.cached`   | Repeat call (should be near-zero)                      |
| `split.measure`           | `measure.measureString` × N subsets                   |
| `split.themeBuild`        | `buildThemeCSS` shared across N subsets               |

## Interpretation

Ratios `< 1.0` mean the new path is faster than the baseline. Anything above
`1.05` on a hot scenario at large scale is a regression worth investigating.

Variance is reported alongside median. If `stddev / median > 0.5` the result
is noisy — increase sample count for that scenario in `run-bench.ts`.

Bench numbers are not stable across machines/CPUs — diff only against
baselines captured on the same machine.
