# Column Schema

Single source of truth for column-type metadata + extension surface.

The runtime, editor, codegen, R defaults, future plugin API — all read
from here. Built up across Phases 1-3; the renderer/behavior pipeline
plugs in during Phase 7.

## Mental model

A column type is a **schema** — a plain TS object describing what
options it has, what it inherits from, what bucket on the wire its
options live in, etc.

Two flavors:

- **Abstract** schemas (BASE, future VIZ, future SVG) — structural /
  ontological. Declare option surfaces inherited by descendants;
  never appear in the column-type picker. `abstract: true`.
- **Concrete** schemas (TEXT, NUMERIC, PERCENT, INTERVAL, …) — what
  users pick. Have `bucket`, `type`, `slots`.

Both kinds compose via `inherits` (single key or array — multi-
inheritance is real: RING ← [PICTOGRAM, PERCENT]). The resolver topo-
sorts the DAG so ancestors render before descendants.

## What lives on a schema

A `ColumnSchema` declares:

- `key`, `label`, `category` — identification
- `inherits` — DAG parent(s)
- `bucket` (concrete only) — wire location of options
- `type` (concrete only) — wire `column.type`
- `slots` (concrete only) — data field selectors (point/lower/upper, …)
- `options[]` — editor knobs (formatting, behavior, styling)
- `optionOverrides` — adjust an inherited option's default

Bundled with the schema (registered alongside via `registerColumnType`)
are:

- **Renderer** (`CellFormatter`) — `(value, opts, ctx, parents) →
  RenderNode`. Pure function returning a render-description tree.
  Browser + SVG-export both consume the same tree.
- **Lifecycle hooks** (`SchemaLifecycle`) — `onFirstPresent` /
  `onLastRemoved` (widget-level: axis, legend) and `onColumnCreate` /
  `onColumnDestroy` (column-level: per-instance handlers).
- **Behaviors** (`SchemaBehaviors`) — type-dispatched logic that used
  to live in scattered switch statements: `sortKey`, `estimateWidth`,
  `emitSource`, `searchKey`, `tooltipText`, `aggregate`.

All four can call `parents.x()` to delegate to ancestors.

Generated by `srcjs/scripts/regenerate-schema.ts`:

- `srcjs/src/types/column-options.generated.ts` — flat per-bucket TS interfaces
- `R/schema-defaults.R` — effective defaults per concrete schema (R-side reads)

CI gate: re-run the script, assert no diff.

## Extension story (third-party schemas)

```ts
import {
  defineSchema, registerColumnType, compose,
} from "@tabviz/core/extend";

const FANCY_RING = defineSchema({
  key: "fancy_ring",
  label: "Fancy Ring",
  inherits: ["ring"],
  options: [
    { key: "innerRadius", control: "number", default: 0.5 },
  ],
});

registerColumnType({
  schema: FANCY_RING,
  renderer: (val, opts, ctx, parents) => {
    const baseRing = parents.ring(val, opts, ctx);
    const cutout = addInnerCutout(baseRing, opts.innerRadius);
    const label = parents.percent(val, opts, ctx);
    return compose(cutout, label, { sep: "" });
  },
  behaviors: {
    sortKey: (val, opts, ctx, parents) =>
      parents.ring.sortKey(val, opts, ctx),  // delegate
  },
  lifecycle: {
    onColumnCreate: (column, widget) => {
      const cleanup = attachHoverPulse(column, widget);
      return cleanup;
    },
  },
});
```

The schema entries are stored on the runtime registry; the editor,
renderer, sort, etc. all dispatch through schema key. No core fork
needed.

## Composition primitives

For combining parent outputs in a child renderer:

```ts
compose(point, lower, upper, { sep: ", ", bracketStart: 2 })
//  → "point (lower, upper)"
//  → RenderGroup with children laid out row-wise

compose(value, sd, { sep: " ± ", minor: 2 })
//  → "value ± sd" — sd rendered in smaller, muted secondary text
//  → RenderGroup with the SD child carrying minor TextStyle

compose(ringSvg, percentText, { sep: "" })
//  → SVG ring + adjacent text label
//  → RenderGroup mixing SVG and text children
```

`compose` is pure; never mutates inputs; applies bracketing and minor-
styling as a final pass on the assembled children.

## Migration notes

The schemas already exist for ~14 concrete column types. The renderers,
sort, width, source-emit, etc. are still hand-written, dispatching on
`column.type` via switch statements in:

- `lib/filter-sort-utils.ts::sortValueFor`
- `lib/width-utils.ts::estimateColumnWidth`
- `lib/source-emit.ts::emitTypeSpecificArgs`
- `svelte/ForestPlot.svelte` (cell renderers, mid-file)
- `export/svg-generator.ts` (parallel cell render paths)
- `stores/forestStore.svelte.ts`, `stores/slices/layout-zoom.svelte.ts`
- `components/controls/ColumnEditorPopover.svelte` (being replaced)

**Phase 7 cuts these over to schema dispatch.** Today is the contract;
tomorrow is the migration. The cost is real — much of the rendering
pipeline gets rewritten — but the win is real too: every type-dispatched
behavior lives next to the schema that owns it, and adding a new type
becomes one self-contained PR.
