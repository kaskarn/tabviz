S7 Architecture

Internal class system design using R’s S7 object-oriented framework

A tour of the internal S7 class graph behind every tabviz() call. Read this if you want to extend the package, write a custom modifier, or follow the JSON that ends up on the JavaScript side.

Overview

tabviz uses S7, R’s modern object-oriented system, for its internal data structures. S7 provides:

  • Type safety — Properties are validated at construction time
  • Composition — Complex objects built from simpler components
  • Method dispatch — Generic functions that work across types
  • Immutability by default — Modifications create new objects

Class Hierarchy

Core Classes (R/classes-core.R)

Class Purpose
WebSpec Root specification — every field of tabviz() lands somewhere on this object
GroupSpec Row grouping definition with optional nesting
EffectSpec Per-effect spec for multi-effect viz columns (point + lower + upper + style)
GroupSummary Aggregate statistics for a group (pooled estimate diamond)
PlotLabels Title, subtitle, caption, footnote text
PaginateSpec Pagination configuration (rows per page, break_on, orphan_min, …); see Pagination
SplitForest A tabviz(split_by = ...) result — named list of per-subview WebSpecs plus the navigation tree

Theme Classes (R/classes-theme.R)

The theme system is resolved through three tiers. Tier 1 is what authors edit; Tier 2 is the chrome / data role bindings derived from Tier 1; Tier 3 is the per-component bundles the renderer actually paints.

Tier Classes Purpose
1 — inputs ThemeInputs, ThemeVariants The 16 customer-facing inputs (primary, secondary, accent, status_*, series_anchors, slot_style, fonts) plus three per-table variants (density, header_style, first_column_style).
2 — chrome roles Surfaces, Content, Dividers, AccentRoles, StatusColors, Semantics Derived from chrome inputs (neutrals + primary + accent). Drives surface tones, body text, dividers, status colors, semantic-token base colors.
2 — data roles SlotRole (per series anchor) Derived from data inputs (series_anchors + slot_style). Each anchor expands into a 7-field bundle: fill, stroke, fill_muted, stroke_muted, fill_emphasis, stroke_emphasis, text_fg.
2 — typography & spacing TextRole, TextRoles, SpacingTokens 8 named text roles; 12 density-derived numeric tokens.
3 — component bundles HeaderCluster / HeaderVariant (light/tint/bold), ColumnGroupCluster, RowGroupCluster / RowGroupTier, RowCluster (base/alt/hover/selected + 5 RowSemantic bundles emphasis/muted/accent/bold/fill), CellCluster, FirstColumnCluster / FirstColumnVariant, PlotScaffold, MarksRecipes / MarkRecipe, AnnotationCluster Component-by-component bindings the renderer reads at paint time.
config AxisConfig, Layout Forest x-axis range / ticks / gridlines / etc.; layout-level plot width and container border.

The composition root is WebTheme; see Theming Manifesto for the why behind this shape.

Component Classes (R/classes-components.R)

Class Purpose
ColumnSpec Single column definition (data + visualization columns alike)
ColumnGroup Hierarchical column header grouping
InteractionSpec UI interaction toggles (sort, collapse, select, edit, themes, etc.)
VizBarEffect, VizBoxplotEffect, VizViolinEffect Per-effect specs for the corresponding viz_*() columns

Annotation Classes (R/classes-annotations.R)

Class Purpose
ReferenceLine Vertical reference line at a specific x value (refline())
CustomAnnotation Shape annotation tied to specific rows (forest_annotation(), frontend WIP)

Design Patterns

1. Class + Helper Function Pattern

Every S7 class has a corresponding helper function with a user-friendly API:

# S7 class (internal)
GroupSpec <- new_class("GroupSpec", properties = list(
  id = class_character,
  label = class_character,
  collapsed = new_property(class_logical, default = FALSE),
  parent_id = new_property(class_character, default = NA_character_)
))

# Helper function (user-facing)
web_group <- function(id, label = id, parent = NULL, collapsed = FALSE) {
  GroupSpec(
    id = as.character(id),
    label = as.character(label),
    collapsed = collapsed,
    parent_id = if (is.null(parent)) NA_character_ else as.character(parent)
  )
}

This pattern provides:

  • Type coercion — Helper handles NULL → NA conversion
  • Defaults — Sensible defaults without cluttering class definition
  • Documentation — roxygen docs on the helper, not the class
  • Validation — Additional business logic before construction

2. Composition Over Inheritance

WebTheme composes ~20 sub-classes across the three theme tiers rather than inheriting or flattening:

WebTheme <- new_class("WebTheme", properties = list(
  name = ...,
  inputs   = ThemeInputs(),     # T1 — what authors edit
  variants = ThemeVariants(),   # T1 — per-table toggles
  surface  = Surfaces(),        # T2 — chrome roles, derived
  content  = Content(),         # T2
  divider  = Dividers(),        # T2
  accent   = AccentRoles(),     # T2
  status   = StatusColors(),    # T2
  semantic = Semantics(),       # T2 — named token color slots
  series   = list(SlotRole()),# T2 — per-series-anchor bundles
  text     = TextRoles(),       # T2 — 8 text roles
  spacing  = SpacingTokens(),   # T2 — density-derived numerics
  header / row_group / row / cell / first_column / plot / ...,  # T3 component bundles
  axis     = AxisConfig(),      # config
  layout   = Layout()           # config
))

Benefits:

  • Targeted modification — Change only what you need: theme@inputs@primary, theme@row@accent@marker_fill, theme@series[[1]]@stroke.
  • Pure derivation — resolve_theme() is idempotent; given the same Tier 1 inputs you get the same Tier 2 / Tier 3 trees, every time.
  • Surgical overrides — set_theme_field() walks the tree by path and pins a single leaf without disturbing siblings; subsequent set_inputs() re-resolves but preserves explicit pins.

3. Fluent Modifiers

Theme modification uses pipe-friendly functions that return modified copies. Each modifier re-resolves the theme before returning, so derived fields stay consistent with explicit overrides.

web_theme_nejm() |>
  set_inputs(brand = "#0066cc") |>
  set_density("comfortable") |>
  set_spacing(row_height = 24) |>
  set_theme_field(c("axis", "gridlines"), TRUE)

The four core theme modifiers:

Modifier Edits
set_inputs() Tier 1 customer inputs (primary / secondary / accent and their _deep companions, status_*, series_anchors, slot_style, fonts)
set_density() set_header_style() set_first_column_style() Per-table variant setters (density, header style, first-column style)
set_spacing() Per-token spacing override on top of the density preset
set_theme_field(theme, path, value) Generic deep setter; path is a character/integer vector walking the theme tree

4. NA vs NULL Convention

The package follows a consistent pattern for optional values:

Type Default Meaning
Scalar optional (string, number) NA_character_ / NA_real_ “Not specified”
Complex object optional NULL “Not present”

  watermark_opacity = NA_real_,

  row_emphasis_col, row_muted_col, row_accent_col, row_fill_col,

  marker_color_col, marker_shape_col, marker_size_col, marker_opacity_col,

Row styling column mappings

Row-level styling stores the column name containing the values, not the values themselves. The five semantic tokens (row_emphasis, row_muted, row_accent, row_bold, row_fill) map to the five RowSemantic bundles on theme@row — naming “this row is special” rather than picking literal colors:

tabviz(data,
  label = "study",
  columns = list(viz_forest(point = "hr", lower = "lo", upper = "hi")),
  row_bold     = "is_summary",       # bare column name
  row_accent   = ~ pval < 0.05,      # or a formula
  row_badge    = "significance",
  row_fill     = "is_pooled"         # 5th semantic token
)

Formulas are evaluated against the data frame at construction time and reduced to logical / character vectors stored under the matching row_*_col slot. This pattern keeps data and configuration separate, supports dynamic per-row styling from data columns, and avoids duplicating data in the spec.

Pagination

When paginate = paginate_spec(...) is set, the breakpoints are computed once R-side over the same flattened display rows the frontend builds, then shipped on the wire as 0-based row ranges (spec.paginate.pages). The Svelte viewer slices displayRows by those ranges and the PDF export consumes the same ranges — drift between viewer and static export is structurally impossible. See Data Flow for the wire format.

Type-Specific Options

The options property holds type-specific configuration:

# Bar column
col_bar("weight", max_value = 100, show_label = TRUE)
# Creates: options = list(bar = list(maxValue = 100, showLabel = TRUE))

# P-value column
col_pvalue("pval", stars = TRUE, thresholds = c(0.05, 0.01, 0.001))
# Creates: options = list(pvalue = list(stars = TRUE, thresholds = ...))

# Forest column
viz_forest(point = "hr", lower = "lo", upper = "hi", scale = "log")
# Creates: options = list(forest = list(point = "hr", scale = "log", ...))

Method Dispatch

S7 methods are registered for generic functions:

# Print method for WebSpec
method(print, WebSpec) <- function(x, ...) {
  cli_inform(c(
    "A {.cls WebSpec} with {nrow(x@data)} row{?s}",
    "*" = "Columns: {length(x@columns)}",
    "*" = "Groups: {length(x@groups)}"
  ))
  invisible(x)
}

# Plot method (renders as widget)
method(plot, WebSpec) <- function(x, ...) {
  forest_plot(x, ...)
}

# as.data.frame method
method(as.data.frame, WebSpec) <- function(x, ...) {
  x@data
}

knit_print for WebSpec / SplitForest

S7 class strings carry the pkg::Class prefix (e.g. "tabviz::WebSpec"), which can’t appear in R function names — so the standard knit_print.tabviz::SplitForest <- pattern is illegal and @exportS3Method doesn’t help. Instead, plain handlers in R/forest_plot.R are bound to the raw class strings via registerS3method() in .onLoad:

# R/forest_plot.R
knit_print_splitforest <- function(x, ...) {
  knitr::knit_print(forest_plot_split(x), ...)
}
knit_print_webspec <- function(x, ...) {
  knitr::knit_print(render_tabviz_widget(x), ...)
}

# R/zzz.R — registered at package load time
registerS3method("knit_print", "tabviz::SplitForest",
                 knit_print_splitforest, envir = asNamespace("knitr"))

Without this binding, knitr’s auto-print fell back to S7’s default print on a chunk whose last value was a WebSpec or SplitForest — which dumped the object’s internal structure as raw <pre> text. The registered handlers route through forest_plot_split() / render_tabviz_widget() so chunks render as live widgets.

Best Practices

Creating New Classes

  1. Start with the helper function API, then design the class to support it
  2. Use NA_* for optional scalars, NULL for optional objects
  3. Add validators for invariants that can’t be expressed as types
  4. Document the helper, not the class directly

Modifying Existing Classes

  1. Add new properties with defaults to maintain backward compatibility
  2. Update serialization in utils-serialize.R
  3. Update TypeScript types in srcjs/src/types/index.ts
  4. Update fluent modifiers if the property should be user-configurable

```r
  style_color = new_property(class_character, default = NA_character_),
  ...
))
```

### Type-Specific Options

The `options` property holds type-specific configuration:

```r
# Bar column
col_bar("weight", max_value = 100, show_label = TRUE)
# Creates: options = list(bar = list(maxValue = 100, showLabel = TRUE))

# P-value column
col_pvalue("pval", stars = TRUE, thresholds = c(0.05, 0.01, 0.001))
# Creates: options = list(pvalue = list(stars = TRUE, thresholds = ...))

# Forest column
viz_forest(point = "hr", lower = "lo", upper = "hi", scale = "log")
# Creates: options = list(forest = list(point = "hr", scale = "log", ...))
```

## Method Dispatch

S7 methods are registered for generic functions:

```r
# Print method for WebSpec
method(print, WebSpec) <- function(x, ...) {
  cli_inform(c(
    "A {.cls WebSpec} with {nrow(x@data)} row{?s}",
    "*" = "Columns: {length(x@columns)}",
    "*" = "Groups: {length(x@groups)}"
  ))
  invisible(x)
}

# Plot method (renders as widget)
method(plot, WebSpec) <- function(x, ...) {
  forest_plot(x, ...)
}

# as.data.frame method
method(as.data.frame, WebSpec) <- function(x, ...) {
  x@data
}
```

### knit_print for WebSpec / SplitForest

S7 class strings carry the `pkg::Class` prefix (e.g. `"tabviz::WebSpec"`),
which can't appear in R function names — so the standard
`knit_print.tabviz::SplitForest <-` pattern is illegal and `@exportS3Method`
doesn't help. Instead, plain handlers in `R/forest_plot.R` are bound to
the raw class strings via `registerS3method()` in `.onLoad`:

```r
# R/forest_plot.R
knit_print_splitforest <- function(x, ...) {
  knitr::knit_print(forest_plot_split(x), ...)
}
knit_print_webspec <- function(x, ...) {
  knitr::knit_print(render_tabviz_widget(x), ...)
}

# R/zzz.R — registered at package load time
registerS3method("knit_print", "tabviz::SplitForest",
                 knit_print_splitforest, envir = asNamespace("knitr"))
```

Without this binding, knitr's auto-print fell back to S7's default print
on a chunk whose last value was a `WebSpec` or `SplitForest` — which
dumped the object's internal structure as raw `<pre>` text. The
registered handlers route through `forest_plot_split()` /
`render_tabviz_widget()` so chunks render as live widgets.

## Best Practices

### Creating New Classes

1. **Start with the helper function API**, then design the class to support it
2. **Use `NA_*` for optional scalars**, `NULL` for optional objects
3. **Add validators** for invariants that can't be expressed as types
4. **Document the helper**, not the class directly

### Modifying Existing Classes

1. **Add new properties with defaults** to maintain backward compatibility
2. **Update serialization** in `utils-serialize.R`
3. **Update TypeScript types** in `srcjs/src/types/index.ts`
4. **Update fluent modifiers** if the property should be user-configurable
